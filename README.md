# RunnerTray

RunnerTray is a native macOS 14+ menu bar app for managing a single local GitHub Actions self-hosted runner entirely from the GUI.

## What it does

- Adds a persistent menu bar icon using `NSStatusItem`.
- Manages one runner through a per-user LaunchAgent at `~/Library/LaunchAgents/com.runnertray.github-runner.plist`.
- Supports Start, Stop, Restart, Pause After Current Job (drain mode), and Resume.
- Exposes status transitions: Not Configured, Stopped, Starting, Idle, Busy, Pausing, Error.
- Provides Settings and Log Viewer windows built with SwiftUI.
- Runs `config.sh` from the app when configuring a runner.

## v1 limitations

- Single runner configuration only.
- No privileged helper, no LaunchDaemon mode, no root operations.
- No automatic runner binary download.
- No direct GitHub API integration.
- No code-signing/notarization workflow in this repository.

## Project structure

- `RunnerTray/RunnerTrayApp.swift` – app entrypoint and window coordination.
- `RunnerTray/MenuBar/` – AppKit status item and menu handling.
- `RunnerTray/Models/` – status and settings models.
- `RunnerTray/Services/` – config, launchctl, status detection, logs, keychain, login item.
- `RunnerTray/ViewModels/` – observable app state and action orchestration.
- `RunnerTray/Views/` – Settings and Log Viewer SwiftUI screens.
- `RunnerTray/Utilities/` – shared process execution utility.

## How status is detected

RunnerTray uses layered heuristics:

1. LaunchAgent state from `launchctl print gui/<uid>/com.runnertray.github-runner`.
2. Runner listener process check (`pgrep -f Runner.Listener`).
3. Worker process check (`pgrep -f Runner.Worker`) for busy detection.
4. `_diag` log tail pattern matching for recent started/finished/error hints.
5. Fallback to `Idle` when listener is active and no active work is detected.

This detection logic is isolated in `RunnerStatusService` so it can be improved later.

## Using an existing runner folder

1. Open **RunnerTray**.
2. Open **Settings** from the menu bar.
3. Enable **Use Existing Runner Folder**.
4. Choose the folder containing your existing runner installation (must include `run.sh` and usually `config.sh`).
5. Save settings.
6. Use **Start Runner** in the menu.

## Configuring a new runner from the app

1. Download/extract GitHub’s runner package manually into a target folder.
2. In **Settings**, fill:
   - GitHub URL (`https://github.com/org/repo`, org, or enterprise URL)
   - Runner Name
   - Labels
   - Work Folder
   - Runner Installation Folder
   - Registration Token
3. Click **Configure Runner**.
4. RunnerTray invokes `config.sh` via `Process`.
5. The token field is cleared after successful configuration.

> Sensitive values are never printed to app logs. The registration token is written to Keychain if saved during configuration.

## Pause behavior (drain mode)

`Pause After Current Job` means:

- If Busy: RunnerTray enters **Pausing**, polls status, then stops LaunchAgent once idle.
- If already Idle: RunnerTray stops immediately.
- No in-progress job is force-killed by pause mode.

Use **Resume** to start the LaunchAgent again.

## Launch at login

RunnerTray uses modern ServiceManagement API (`SMAppService.mainApp`) to toggle launch-at-login for the app itself.

## Settings persistence and secrets

- Non-sensitive settings: `UserDefaults` JSON blob (`runnertray.settings`).
- Sensitive token data: Keychain (`com.runnertray.app` service namespace).

## Logs

- Runner diagnostics are read from `<runner folder>/_diag`.
- Log Viewer displays latest log file with manual refresh.
- Settings view includes compact recent log tail (40 lines).
- **Open Logs** / **Reveal in Finder** opens the diagnostics folder.

## Build and run in Xcode

1. Open `RunnerTray.xcodeproj` in Xcode 15+ on macOS 14+.
2. Select the **RunnerTray** target.
3. Build and run.
4. The app runs as a menu bar utility (`LSUIElement = true`) without a Dock icon.

## Permissions / user actions you may need

- If macOS prompts for login item permissions, approve as needed.
- Ensure selected runner folder is readable/writable by your user.
- If runner scripts lack execute permission, run `chmod +x config.sh run.sh` in that folder once.

## LaunchAgent plist example generated by RunnerTray

RunnerTray writes an XML plist with values equivalent to:

- Label: `com.runnertray.github-runner`
- ProgramArguments: `/bin/bash -lc ./run.sh`
- WorkingDirectory: `<runner folder>`
- RunAtLoad: `true`
- KeepAlive: `true`
- StandardOutPath: `<runner folder>/runnertray-stdout.log`
- StandardErrorPath: `<runner folder>/runnertray-stderr.log`

## Future extension notes (multi-runner)

The architecture is prepared for expansion by:

- Keeping runner status heuristics in `RunnerStatusService`.
- Encapsulating launchctl behavior in `LaunchAgentService`.
- Storing settings in a codable model that can evolve from singleton to array.
- Isolating UI orchestration in `AppViewModel`, making it feasible to introduce per-runner view models.
